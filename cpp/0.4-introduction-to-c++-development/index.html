軟件開發過程

第一步:定義你想要解決的問題

這是what步驟，在這裡你要弄清楚你打算解決什麼問題。
提出你想要編程的最初想法可能是最簡單的步驟，也可能是最難的步驟。
但從概念上講，它是最簡單的。
您所需要的只是一個可以明確定義的想法，您就可以為下一步做好準備。

這裡有一些例子：

  • “我想編寫一個程序，允許我輸入許多數字，然後計算平均值。”
  • “我想編寫一個生成二維迷宮並讓用戶在其中導航的程序。如果他們到達終點，用戶就贏了。”
  • “我想編寫一個程序來讀取股票價格文件並預測股票是上漲還是下跌。”
第 2 步：確定您將如何解決問題

這是“如何”步驟，您可以在此處確定如何解決您在步驟 1 中遇到的問題。這也是軟件開發中最容易被忽視的步驟。
問題的癥結在於解決問題的方法有很多種——然而，這些解決方案有的好，有的不好。
很多時候，程序員會想到一個想法，坐下來，然後立即開始編寫解決方案。
這通常會生成屬於不良類別的解決方案。
通常，好的解決方案具有以下特徵：

  • 它們簡單明了（不過分複雜或令人困惑）。
  • 它們有據可查（尤其是圍繞所做的任何假設或限制）。
  • 它們是模塊化構建的，因此部分可以在以後重複使用或更改，而不會影響程序的其他部分。
  • 它們很健壯，可以在意外發生時恢復或給出有用的錯誤信息。

當您坐下來立即開始編碼時，您通常會想“我想做 ”，因此您會實施能夠最快實現目標的解決方案。
這可能會導致程序變得脆弱、以後難以更改或擴展，或者有很多錯誤（技術缺陷）。

順便一提…
早在 1870 年代，Thomas Edison 就首次使用了 bug 這個術語！然而，這個詞在 1940 年代流行起來，當時工程師發現一隻真正的飛蛾卡在早期計算機的硬件中，導致短路。
報告錯誤的日誌和飛蛾現在都屬於史密森尼美國歷史博物館。
可以在這裡查看。

研究表明，程序員只有 20% 的時間實際用於編寫初始程序。
另外 80% 用於維護，包括調試（消除錯誤）、更新以應對環境變化（例如在新的操作系統版本上運行）、增強（小的變化以提高可用性或功能），或內部改進（以提高可靠性或可維護性）。
因此，在開始編寫代碼之前，花一點額外的時間來思考解決問題的最佳方法、您所做的假設以及您如何為未來做計劃是值得的，這樣可以為您節省大量的時間和麻煩。
我們將在以後的課程中討論如何有效地設計問題的解決方案。

步驟3:編寫程序

為了編寫程序，我們需要兩件事：首先，我們需要了解編程語言——這就是這些教程的目的！其次，我們需要一個文本編輯器來編寫和保存我們寫好的程序。
我們使用 C++ 指令編寫的程序稱為源代碼（通常簡稱為代碼）。
可以使用您想要的任何文本編輯器編寫程序，甚至是像 Windows 的記事本或 Unix 的 vi 或 pico 這樣簡單的編輯器。
但是，我們強烈建議您使用專為編程設計的編輯器（稱為代碼編輯器）。
如果您還沒有，請不要擔心。
我們將很快介紹如何安裝代碼編輯器。
為編碼而設計的典型編輯器具有一些使編程更容易的功能，包括：

  1. 行編號。當編譯器給我們一個錯誤時，行號很有用，因為典型的編譯器錯誤會指出：一些錯誤代碼/消息，第 64 行。如果沒有顯示行號的編輯器，找到第 64 行可能會很麻煩。
    2.語法高亮和著色。語法突出顯示和著色會更改程序各個部分的顏色，以便更容易識別程序的不同組件。下面是一個帶有行號和語法高亮的 C++ 程序示例：

#include <iostream>

int main()
{
    std::cout << "Colored text!";
    return 0;

我們在本教程中展示的示例總是有行編號和語法高亮顯示，以使它們更容易理解。

  1. 明確的字體。非編程字體通常很難區分數字 0 和字母 O，或者區分數字 1、字母 l（小寫 L）和字母 I（大寫 i）。一種好的編程字體將確保這些符號在視覺上有所區別，以確保不會意外地用一個符號代替另一個符號。默認情況下，所有代碼編輯器都應啟用此功能，但標准文本編輯器可能不會。

您編寫的程序通常被命名為 something.cpp，其中的某些內容會替換為您為程序選擇的名稱（例如計算器、hi-lo 等）。 .cpp 擴展名告訴編譯器（和您）這是一個包含 C++ 指令的 C++ 源代碼文件。請注意，有些人使用擴展名 .cc 而不是 .cpp，但我們建議您使用 .cpp。

最佳實踐

將您的代碼文件命名為 something.cpp，其中 something 是您選擇的名稱，而 .cpp 是表明該文件是 C++ 源文件的擴展名。
另請注意，許多複雜的 C++ 程序都有多個 .cpp 文件。儘管您最初要創建的大多數程序只有一個 .cpp 文件，但可以編寫具有數十個或數百個 .cpp 文件的單個程序。
一旦我們編寫了我們的程序，接下來的步驟就是將源代碼轉換成我們可以運行的東西，然後看看它是否有效！我們將在下一課討論這些步驟 (4-7)。
